# i18n Implementation Notes

## Architecture Overview

- **Library**: Paraglide JS (`@inlang/paraglide-js`)
- **Strategy**: `["baseLocale"]` only — Paraglide never reads cookies
- **Cookie**: single `locale` cookie managed outside Paraglide (via `hooks.server.js` and `locale.svelte.js`)
- **Locale priority**: `locale` cookie → user DB preference → baseLocale (`en`)

## Server-side locale override (AsyncLocalStorage nesting)

Paraglide middleware determines locale from its strategy (`["baseLocale"]` → always `'en'`).
We override this via nesting a second `serverAsyncLocalStorage.run()` call inside the
middleware callback:

```
authHandle: reads locale cookie → event.locals.effectiveLocale = 'fr'
    ↓
paraglideHandle: middleware runs → locale = 'en' (baseLocale)
    ↓
middleware callback: serverAsyncLocalStorage.run({ locale: 'fr' }, () => resolve())
    ↓ getLocale()
locale: 'fr' ← messages render in French
```

This is per-request safe: AsyncLocalStorage is isolated per request, no race condition.

**File**: `src/hooks.server.js` — `paraglideHandle` function.

## Cookie management

### The httpOnly trap

SvelteKit's `event.cookies.set()` defaults to `httpOnly: true`. This means JavaScript
cannot read or write the cookie via `document.cookie`. All cookies set server-side
MUST explicitly pass `httpOnly: false` to be accessible from the client:

```js
event.cookies.set("locale", "fr", {
  path: "/",
  maxAge: 34560000,
  sameSite: "lax",
  httpOnly: false, // ← required for JS access
});
```

### Client-side cookie setting

The `Locale` class in `src/lib/locale.svelte.js` sets the `locale` cookie via
`document.cookie` on every `setLocale()` call. This works because:

- The cookie was initially set by the server with `httpOnly: false`
- Subsequent `document.cookie` writes work normally

### Footer switcher

Clicking "Français" in the footer calls `setLocale('fr')` → `overwriteSetLocale`
sets `locale=fr` via `document.cookie` → `window.location.reload()`.

### Login flows

All three login flows (password, magic link, OIDC) force the `locale` cookie to the
user's database preference after creating the session.

**Files**: `login/+page.server.js` (signIn action), `magic-login/callback/+page.server.js`,
`login/oidc/callback/+server.js`.

## Email templates

Located in `src/lib/backend/email/`. Format: YAML with `{{ link }}` Jinja-like interpolation.

```
src/lib/backend/email/
├── magic-link.yaml
├── reset-password.yaml
└── index.js
```

`renderEmail(name, { link, locale })` reads the YAML file via `js-yaml`, selects the
correct locale (fallback `en`), and replaces `{{ link }}` with the actual URL.

Locale is queried from the database using the recipient's email address.

## Adding a new page with i18n

1. Add keys to `messages/en.json` and `messages/fr.json`
2. In the Svelte component: `import { m } from "$lib/paraglide/messages"`
3. Replace hardcoded strings with `{m.your_key()}`
4. Run `pnpm build` to regenerate compiled messages
